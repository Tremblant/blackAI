Layer 1 — Hardware
Linux Kernel + GPU Device
lspci | grep -i nvidia lsmod | grep nvidia
The GPU must be visible to the PCI bus and the kernel module must be loaded. This layer is passive — it either works or it doesn't. If lspci shows nothing, it's a hardware or BIOS issue. If lsmod shows nothing, the driver isn't installed.
Layer 2 — Driver
nvidia-driver-470
ubuntu-drivers devices sudo apt install nvidia-driver-470 sudo reboot nvidia-smi
The driver is the bridge between OS and GPU silicon. Version must match GPU architecture — GT 710 is Kepler, which tops out at driver 470 and CUDA 11.4. Wrong driver = silent failure or kernel panic.
Layer 3 — Container Runtime
NVIDIA Container Toolkit
sudo apt install -y nvidia-container-toolkit sudo nvidia-ctk runtime configure --runtime=docker sudo systemctl restart docker
The toolkit wraps Docker's runtime to intercept GPU requests. Without it, --gpus all is silently ignored. This layer also exposes GPU device nodes into the container namespace.
Layer 4 — Application
PyTorch + CUDA + Docker
docker run --rm --gpus all \ nvidia/cuda:12.1.1-base-ubuntu22.04 \ nvidia-smi
The final validation: GPU visible inside a container. When this works, the entire stack is operational. PyTorch can call .cuda(), models can be moved to GPU memory, and training infrastructure is ready.
